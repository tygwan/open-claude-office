# 개발 전주기: 기술 분야별 상세 가이드

소프트웨어 개발 전주기를 **기술 분야(Domain)** 관점에서 정리합니다. 각 분야가 10-Stage 개발 과정에서 언제, 어떤 활동을, 어떤 도구로 수행하는지 명세합니다.

> 참고: Stage별 프로세스/관리 관점은 `DEVELOPMENT-LIFECYCLE.md` 참조
> 참고: MVP/PoC/Production 단계별 적용은 `MVP-POC개발전주기.md` 참조

---

```yaml

  title: Project API
  version: 1.0.0
paths:
  /api/v1/users:

      summary: List users
      parameters:
        - name: page
          in: query
          schema: { type: integer, default: 1 }
      responses:
        '200':
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/UserListResponse'
  /api/v1/users/{id}:
    get:
      summary: Get user by ID
      parameters:
        - name: id
          in: path
          required: true
          schema: { type: string, format: uuid }
      responses:
        '200': { description: Success }
        '404': { description: Not found }
components:
  schemas:
    User:
      type: object
      required: [id, email, name]
      properties:
        id: { type: string, format: uuid }
        email: { type: string, format: email }
        name: { type: string }
  securitySchemes:
    bearerAuth: { type: http, scheme: bearer, bearerFormat: JWT }
```

## 3. Database

### 역할
데이터의 영구 저장, 조회, 무결성 보장, 관계 관리

### 세부 분야

| 분야 | 설명 | 주요 기술 |
|------|------|----------|
| RDBMS | 관계형 데이터베이스 | PostgreSQL, MySQL, MariaDB, SQLite |
| NoSQL — Document | 문서형 | MongoDB, Firestore |
| NoSQL — Key-Value | 키-값 (캐시, 세션) | Redis, DynamoDB |
| NoSQL — Graph | 그래프 관계 | Neo4j, ArangoDB |
| NoSQL — Time Series | 시계열 데이터 | InfluxDB, TimescaleDB |
| Vector DB | 임베딩 유사도 검색 | Pinecone, Weaviate, pgvector |
| ORM/Query Builder | 코드-DB 매핑 | Prisma, TypeORM, Drizzle, SQLAlchemy |
| Migration | 스키마 버전 관리 | Prisma Migrate, Alembic, Flyway |
| 스키마 설계 | ERD, 정규화 | dbdiagram.io, pgModeler |
| 인덱싱 | 쿼리 성능 최적화 | B-tree, GIN, GiST, Partial Index |
| 백업/복구 | 데이터 보호 | pg_dump, WAL, PITR |
| 복제 | 고가용성 | Primary-Replica, Multi-Master |
| 연결 관리 | Connection Pool | PgBouncer, ProxySQL |

### DB 선택 가이드

```
정형 데이터 + ACID 필요          → PostgreSQL (기본 추천)
빠른 프로토타이핑               → SQLite (MVP) → PostgreSQL (PoC+)
문서 유연성                     → MongoDB
캐시 + 세션 + 큐               → Redis
검색                           → Elasticsearch / Meilisearch
AI/ML 임베딩                   → pgvector (PostgreSQL 확장)
시계열 메트릭                   → TimescaleDB / InfluxDB
대규모 분산 쓰기                → Cassandra / ScyllaDB
```

### Migration 예시 (Prisma)

```prisma
model User {
  id        String   @id @default(uuid())
  email     String   @unique
  name      String
  role      Role     @default(USER)
  posts     Post[]
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  @@index([email])
}

model Post {
  id        String   @id @default(uuid())
  title     String
  content   String?
  published Boolean  @default(false)
  author    User     @relation(fields: [authorId], references: [id])
  authorId  String
  @@index([authorId])
  @@index([published, createdAt])
}

enum Role { USER ADMIN MODERATOR }
```

### CI/CD 예시 (GitHub Actions)

```yaml
name: CI
on:
  push: { branches: [main, develop] }
  pull_request: { branches: [main] }
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci && npm run lint && npm run type-check
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env: { POSTGRES_DB: test, POSTGRES_PASSWORD: test }
        ports: ['5432:5432']
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run test:coverage
  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build
  deploy-staging:
    if: github.ref == 'refs/heads/develop'
    needs: build
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - run: echo "Deploy to staging"
  deploy-production:
    if: github.ref == 'refs/heads/main'
    needs: build
    runs-on: ubuntu-latest
    environment: { name: production, url: 'https://app.example.com' }
    steps:
      - run: echo "Deploy to production"
```

### Dockerfile 예시 (Node.js)

```dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine AS runner
WORKDIR /app
RUN addgroup --system --gid 1001 nodejs && \
    adduser --system --uid 1001 appuser
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
USER appuser
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -q --spider http://localhost:3000/health || exit 1
CMD ["node", "dist/main.js"]
```

## 5. Security

### 역할
시스템 전체의 보안을 설계, 구현, 검증, 운영

### 세부 분야

| 분야 | 설명 | 주요 기술 |
|------|------|----------|
| 앱 보안 | OWASP Top 10 방어 | Input validation, CSP, CORS, Helmet |
| 인증 | 사용자 신원 확인 | OAuth2, OIDC, JWT, Passkey, MFA |
| 인가 | 접근 권한 제어 | RBAC, ABAC, Policy Engine |
| SAST | 소스코드 정적 분석 | Snyk, SonarQube, Semgrep, CodeQL |
| DAST | 실행 중 동적 분석 | OWASP ZAP, Burp Suite |
| 의존성 보안 | 취약 패키지 감지 | npm audit, Dependabot, Snyk |
| Secret 관리 | 비밀 정보 보호 | Vault, AWS Secrets Manager |
| Secret 스캔 | 코드 내 비밀 노출 | gitleaks, truffleHog |
| 인프라 보안 | 네트워크, 방화벽 | VPC, WAF, Security Groups |
| 컨테이너 보안 | 이미지 취약점 | Trivy, Snyk Container |
| 암호화 | 데이터 보호 | TLS, AES-256, bcrypt/argon2 |
| 컴플라이언스 | 규정 준수 | SOC2, GDPR, HIPAA |

### OWASP Top 10 대응

| 위협 | 방어 |
|------|------|
| A01 Broken Access Control | RBAC/ABAC, 서버사이드 검증, 최소 권한 |
| A02 Cryptographic Failures | TLS 강제, 민감 데이터 암호화 |
| A03 Injection | Parameterized queries(ORM), Input validation |
| A04 Insecure Design | Threat modeling, Security-by-design |
| A05 Security Misconfiguration | Hardened defaults, Helmet, CSP |
| A06 Vulnerable Components | Dependabot, npm audit |
| A07 Auth Failures | MFA, Rate limiting, Account lockout |
| A08 Data Integrity Failures | SBOM, 서명 검증, CI/CD 보호 |
| A09 Logging Failures | 구조화 로깅, 감사 로그, 알림 |
| A10 SSRF | URL 화이트리스트, 내부 네트워크 차단 |

### Stage별 활동

| Stage | Security 활동 |
|-------|-------------|
| Stage 3 (Design) | 위협 모델링, 인증/인가 전략, 암호화 전략 |
| Stage 4 (Env Setup) | Secret 관리, gitleaks 훅, CSP 설정 |
| Stage 5 (Dev) | Input validation, Parameterized queries, CORS |
| Stage 6 (Test) | SAST(PoC+), DAST(Prod), 의존성 스캔, 침투 테스트 |
| Stage 7 (Deploy) | TLS 설정, WAF 규칙, 보안 그룹 검증 |
| Stage 8 (Monitor) | 감사 로그, 이상 탐지, 보안 알림 |

## 7. Observability

### 역할
시스템 내부 상태를 외부에서 관찰 가능하게 만듦

### 3 Pillars

```
Logs (무엇이 일어났나)
  "ERROR payment_service: charge failed user=123 error=card_declined"

Metrics (얼마나 일어나는가)
  http_requests_total{status="500"} 23
  http_request_duration_seconds{p99} 2.5s

Traces (어디서 느린가)
  Request → API Gateway(2ms) → Auth(15ms) → Payment(2300ms!) → DB(5ms)
```

### 세부 분야

| 분야 | 설명 | 주요 기술 |
|------|------|----------|
| 로그 (Logs) | 이벤트 기록, 구조화 | ELK, Loki, Fluentd, Pino |
| 메트릭 (Metrics) | 수치 지표 수집 | Prometheus, Grafana, Datadog |
| 트레이스 (Traces) | 분산 요청 추적 | Jaeger, OpenTelemetry |
| APM | 앱 성능 통합 | Datadog, New Relic |
| 에러 추적 | 런타임 에러 수집 | Sentry, Bugsnag |
| 알림 | 이상 탐지, 발송 | PagerDuty, OpsGenie, Grafana |
| 상태 페이지 | 서비스 상태 공개 | Statuspage, Upptime |
| 합성 모니터링 | 외부 정기 체크 | Checkly, Pingdom |
| 대시보드 | 시각화 | Grafana, Kibana |

### Stage별 활동

| Stage | Observability 활동 |
|-------|-------------------|
| Stage 4 (Env Setup) | 로깅 라이브러리, 구조화 로그 포맷 |
| Stage 5 (Dev) | 로그 포인트, 커스텀 메트릭 정의 |
| Stage 7 (Deploy) | Health check, Smoke test |
| Stage 8 (Monitor) | APM, 로그 수집, 알림, 대시보드, 인시던트 대응 |
| Stage 9 (Post) | 알림 튜닝, SLO/SLA 모니터링 |

### 인시던트 대응 프로세스

```
1. 감지 (Detect)       알림 수신 → On-call 확인
2. 분류 (Triage)       SEV1(서비스 중단) / SEV2(기능 장애) / SEV3(성능 저하)
3. 대응 (Respond)      SEV1: 즉시 / SEV2: 1시간 / SEV3: 다음 비즈니스 아워
4. 복구 (Recover)      Rollback / Hotfix / 수동 개입
5. 후속 (Follow-up)    Post-mortem (72시간 이내) → LEARNINGS.md 기록
```
